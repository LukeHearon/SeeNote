# Log Interpretation Guide — Intel Mac Video Playback Diagnosis

## Your Task

Read the debug log files in this directory (`debug_video/run_0*.txt`) and determine which of three possible causes is responsible for the reported video playback failures on an Intel MacBook Pro. Then state a confident diagnosis and recommended fix.

The log files were generated by running SeeNote (a Tauri v2 + React/TypeScript desktop app) on an Intel MacBook Pro where:
- Video picture freezes a few seconds after playback starts in **Fast mode** (audio continues; spectrogram continues scrolling)
- Playing a selection in **Mixed mode** either never starts or audio does not play

These failures do not occur in a regular web browser (Firefox, Chrome) on the same machine with the same files.

---

## App Architecture (what matters for diagnosis)

SeeNote is a Tauri app. On macOS, Tauri uses **WKWebView** — Apple's WebKit-based WebView — not Chromium. This is the most important fact for interpreting the results.

**Fast mode** plays video using a bare HTML `<video src="tauri://localhost/...">` element. The Tauri app serves the file via a custom asset protocol. The `VideoElementEngine` class drives the element: it calls `el.play()`, reads `el.currentTime` every animation frame (RAF loop), and fires `onTimeUpdate` to move the spectrogram playhead. Audio comes from the video element itself (not a separate audio pipeline).

**Mixed mode + selection** switches to a two-track system:
- Audio: Rust decoder (via `AudioEngine`) — decodes PCM from the video file server-side
- Video frames: `VideoFrameSource` — fetches the full file as an `ArrayBuffer`, demuxes with mp4box.js, decodes frames with the **WebCodecs** `VideoDecoder` API, draws to a `<canvas>` via `ctx.drawImage(VideoFrame)`

The spectrogram playhead is always driven by whichever engine is active. In Fast mode that is `el.currentTime` from the video element. The fact that the spectrogram continues advancing after the picture freezes tells us `el.currentTime` is still advancing — meaning the video element has not stalled entirely, only its picture output has stopped.

## The Three Candidate Causes

### Cause #1 — Tauri Asset Protocol Does Not Handle HTTP Byte-Range Requests

The `<video>` element needs `206 Partial Content` range responses to buffer ahead in the file. If Tauri's `tauri://localhost/` protocol returns `200` for all requests (or handles `Range:` headers incorrectly), the browser decodes only the initial buffer it received, plays those frames, then stalls waiting for data that never arrives. Audio in the initial buffer plays through; the picture freezes at the last decoded frame; `el.currentTime` stops advancing once the buffer is exhausted — or advances until the end of the buffered segment and then pauses.

**Diagnostic signature:**
- `[video-el] WAITING` or `[video-el] STALLED` events appear in the log around the time of the freeze
- `FREEZE DETECTED` log shows `readyState < 4` (values: 0=HAVE_NOTHING, 1=HAVE_METADATA, 2=HAVE_CURRENT_DATA, 3=HAVE_FUTURE_DATA, 4=HAVE_ENOUGH_DATA)
- `buffered=` range in the FREEZE log shows the buffer ends near `currentTime` (i.e., the element ran out of buffered data)
- Run 2 (Blob URL): fixes the problem — blob URLs don't use HTTP range requests; the browser already has the full file in memory

### Cause #2 — WKWebView GPU Compositing Failure

The `<video>` element decodes audio and video correctly (audio plays, `el.currentTime` advances), but the decoded video texture cannot be composited to the screen. This can happen on older Intel Macs when WKWebView's GPU-accelerated video compositing pipeline stalls or is suspended — possibly due to driver limitations, GPU memory pressure, or an energy-saving policy. The CoreAudio audio pipeline is unaffected because it is entirely separate from GPU compositing.

**Diagnostic signature:**
- **No** `WAITING` or `STALLED` events in the log (the element is not buffering)
- `FREEZE DETECTED` shows `readyState=4` (HAVE_ENOUGH_DATA) — the element has data and believes it is fine
- `buffered=` range extends well past `currentTime` — buffer is healthy
- If RVFC is supported: `RVFC presentedFrames=N (+0 in Xms ≈ 0.0fps)` while `currentTime` is still advancing — frames are not reaching the compositor
- Run 3 (Canvas Mirror): **canvas shows moving video frames** even though the native video layer was dark — confirms GPU compositing is the broken link

### Cause #3 — WebCodecs Preroll Too Slow on This Machine

In Mixed+selection mode, the app awaits `prerollVideo()` before starting `AudioEngine`. `prerollVideo` runs a `VideoDecoder` (WebCodecs) decode pass over all frames in the selection. On an underpowered Intel Mac, software WebCodecs decode can take many seconds per second of video. The user sees the buffering spinner indefinitely and concludes playback does not work.

**Diagnostic signature:**
- Run 4 (Mixed + normal preroll): `[preroll] start X-Ys` log line appears but `[preroll] done in Xms` either never appears or appears after a very long elapsed time (thousands of milliseconds — look at the ms value)
- Run 5 (Mixed + preroll skip): `[debug] preroll skipped` appears and audio **does** play immediately
- The problem is isolated to the canvas/selection path; Fast mode may or may not be separately affected

## How to Read the Log Format

Each line is: `[HH:MM:SS] message`

Key message prefixes:
- `[video-el]` — emitted by `VideoElementEngine`, the `<video>` element transport (Fast mode and Mixed without selection)
- `[preroll]` — emitted by the preroll step before AudioEngine starts (Mixed+selection, Accurate)
- `[debug]` — emitted by the debug toggle actions (blob URL, canvas mirror, preroll skip)
- `[video]` — emitted by `VideoFrameSource` (WebCodecs frame decode pipeline)
- Bare messages — general app events (file open, spectrogram load, etc.)

Key `[video-el]` messages and what they mean:

| Message | Meaning |
|---|---|
| `play start=X end=Y readyState=N networkState=M buffered=A-B` | Playback started; snapshot of element state at that moment |
| `play() rejected: <reason>` | `el.play()` returned a rejected Promise — element refused to play |
| `WAITING (buffering stall)` | Element paused because it ran out of buffered data — **cause #1 signal** |
| `STALLED (network stalled)` | Network fetch for video data stopped — **cause #1 signal** |
| `SUSPEND (browser paused loading)` | Browser voluntarily paused data download |
| `canplay` | Buffer filled enough to resume — usually follows a WAITING event |
| `ERROR code=N` | MediaError: 1=aborted, 2=network, 3=decode error, 4=format unsupported |
| `FREEZE DETECTED: currentTime=X … readyState=N networkState=M buffered=A-B` | RAF loop detected `currentTime` unchanged for ~1 second |
| `freeze cleared — currentTime now X (was stuck for N frames)` | Time resumed advancing after a freeze |
| `RVFC: NOT supported` | `requestVideoFrameCallback` not available in this WebView |
| `RVFC presentedFrames=N (+D in Xms ≈ Yfps)` | Compositor frame delivery rate every 2 seconds |

`readyState` values: 0=HAVE_NOTHING, 1=HAVE_METADATA, 2=HAVE_CURRENT_DATA, 3=HAVE_FUTURE_DATA, 4=HAVE_ENOUGH_DATA  
`networkState` values: 0=EMPTY, 1=IDLE, 2=LOADING, 3=NO_SOURCE

## Step-by-Step Diagnosis

Work through these in order. Each step tells you which file to read and what to look for.

### Step 1 — Check if the video element can play at all

Open `run_01_fast_baseline.txt`.

Find the `[video-el] play start=...` line. Check `readyState` at that moment:
- If `readyState=0` or `readyState=1` at play start: the element had barely loaded — look for a subsequent `ERROR` line. If one exists, note its code.
- If `play() rejected` appears: the element refused to play entirely. Note the rejection reason.
- If neither: the element started playing normally. Continue to Step 2.

### Step 2 — Identify the freeze signature (baseline run)

Still in `run_01_fast_baseline.txt`.

Find the `FREEZE DETECTED` line. Note:
1. The `readyState` value
2. Whether `buffered=` ends near or well past `currentTime`
3. Whether `WAITING` or `STALLED` lines appear **before** the freeze

**Decision:**
- `WAITING` or `STALLED` present → **strong Cause #1 signal**. Confirm in Step 3.
- No `WAITING`/`STALLED`, `readyState=4`, buffer extends well past currentTime → **strong Cause #2 signal**. Confirm in Step 4.
- No `FREEZE DETECTED` at all, and no `WAITING`/`STALLED` → the freeze may not have been captured. Check whether RVFC is reporting `0fps` while there is no freeze — that would still indicate Cause #2.

Check the RVFC lines (if present):
- `(+0 in 2000ms ≈ 0.0fps)` while `currentTime` is advancing → compositor stopped delivering frames (Cause #2).
- Frame rate drops from normal to zero around the time of the freeze → Cause #2.
- Frame rate drops and then `WAITING` fires → Cause #1 (element starved for data, which also stopped frame delivery).

### Step 3 — Confirm or rule out Cause #1 (blob URL run)

Open `run_02_fast_bloburl.txt`.

Read the note the user added at the top: did the freeze occur in this run?

- **No freeze (or freeze occurred much later):** Cause #1 confirmed. The Tauri asset protocol is mishandling byte-range requests. A blob URL — where the entire file is already in memory — bypasses this. **Conclusion: Cause #1. Recommended fix: investigate and patch Tauri's asset protocol handler to return proper `206 Partial Content` responses, or pre-fetch the file and use a blob URL in Fast mode.**

- **Freeze still occurs, same timing:** Cause #1 ruled out. The protocol is not the problem. Proceed to Step 4.

Also check: does `run_02` show `WAITING`/`STALLED` events? If no, and Cause #1 was tentatively flagged in Step 2, this is a contradiction — look more carefully at both logs before concluding.

### Step 4 — Confirm or rule out Cause #2 (canvas mirror run)

Open `run_03_fast_canvasmirror.txt`.

Read the note the user added at the top: did the **canvas** show moving video frames?

Also look in the log for:
- `[debug] canvas-mirror: first ctx.drawImage call` — confirms the canvas loop ran and attempted to draw
- `[video-el] FREEZE DETECTED` — confirms the native element froze during this run too

**If the canvas showed moving frames while the native video was dark (or froze earlier):** Cause #2 confirmed. `ctx.drawImage(video, ...)` routes through a different compositing path than the native `<video>` layer in WKWebView. If drawing frames to a 2D canvas works but the native `<video>` display doesn't, WKWebView's GPU video compositor is the broken link. **Conclusion: Cause #2. Canvas Mirror is itself a viable workaround for Fast mode: keep the `<video>` element alive for audio/decoding, draw frames to a canvas. This could be promoted to a permanent "Fast (Canvas)" mode.**

**If the canvas was also black/frozen, or no `drawImage` log line appeared:** Cause #2 is less certain. If RVFC is available and showed 0fps in both runs, the decode pipeline itself may be stalled — not just compositing. This could mean a more fundamental WebCodecs/hardware-decode issue, or it might still be Cause #1 (no data = no frames to decode). Review the buffer state in the FREEZE DETECTED lines of both runs.

### Step 5 — Diagnose Mixed mode selection playback

Open `run_04_mixed_selection_normal.txt` and `run_05_mixed_selection_skip.txt`.

In `run_04`, find `[preroll] start X-Ys` and `[preroll] done in Nms`:
- If `done in Nms` never appears: preroll hung and never resolved. Cause #3 confirmed.
- If `done in Nms` appears with N > 5000 (5 seconds or more): preroll is extremely slow. Cause #3 confirmed.
- If `done in Nms` appears quickly (< 1000ms): preroll is not the issue. Look for other errors (AudioEngine errors, `[video]` decoder errors).

In `run_05`, check the user's note: did audio play after `[debug] preroll skipped`?
- **Yes, audio played:** Cause #3 confirmed. The WebCodecs decode on this machine is too slow to preroll in a reasonable time.
- **No, audio did not play:** The selection playback problem is not preroll. Look for AudioEngine errors or transport errors in the log.

## Synthesizing a Diagnosis

After working through the steps, you should be able to make one of these diagnoses:

**Diagnosis A — Cause #1 (Tauri protocol):** `WAITING`/`STALLED` in baseline; no freeze (or delayed freeze) in blob URL run. Fix: patch the Tauri asset protocol to handle `Range:` request headers correctly, or switch Fast mode to pre-fetch the file into a blob URL before assigning it to the `<video>` element (the blob URL approach is already proven to work by run 2).

**Diagnosis B — Cause #2 (GPU compositing):** No `WAITING`/`STALLED`; `readyState=4` at freeze; RVFC drops to 0fps; canvas mirror shows frames when native element is dark. Fix: promote Canvas Mirror to a permanent "Fast (Canvas)" rendering mode. The `<video>` element continues decoding audio and video; frames are drawn to a 2D canvas each RAF tick. This is slightly more CPU work than native compositing but far cheaper than WebCodecs and eliminates the compositing dependency.

**Diagnosis C — Cause #3 (WebCodecs preroll):** Fast mode may or may not be affected; Mixed+selection never starts until preroll is skipped; skipping preroll allows audio to play. Fix: add a 2–3 second preroll timeout in `togglePlay` so AudioEngine starts even if WebCodecs hasn't finished decoding. Frames will catch up asynchronously. (The "2s timeout" option in the debug controls tests this directly — if the user tries it and audio plays, this is immediately a viable workaround.)

**Multiple causes possible.** Cause #3 can coexist with Cause #1 or #2 — the Fast mode failure and the Mixed+selection failure are independent code paths. State both if the evidence points to both.

If the evidence is ambiguous or contradictory (e.g., canvas mirror also shows no frames), state what is ruled out, what remains uncertain, and what additional information would resolve it.
